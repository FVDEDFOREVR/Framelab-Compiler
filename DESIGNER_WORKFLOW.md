# Designer Workflow v0.1

This document defines the simplest first designer-facing workflow for FrameLab.

It is a product workflow spec only. It does not define a UI implementation.

The workflow is intentionally scoped to the current supported subset described in [SUPPORTED.md](/Users/tars/Desktop/framelab-compiler/SUPPORTED.md).

## Goal

Give a designer one believable path from design intent to production-shaped UI without asking them to think about compiler phases, ASTs, IR, or diagnostics JSON.

## Workflow Summary

The first workflow is:

1. start from a prompt or structured brief
2. let AI generate a first-pass FrameLab draft
3. preview the result visually
4. refine structure in a familiar design-oriented step
5. hand off production-shaped UI generated by the compiler

This is a narrow workflow for the current subset, not a promise of full-spec design tooling.

## What The Designer Sees

The designer-facing flow should feel like this:

1. Describe the component in plain language or fill in a small structured brief.
2. See a first visual preview of the component.
3. Adjust a few design-facing choices such as:
   - component name
   - text content fields
   - variant values
   - layout structure such as stacked sections or optional slots
   - token-backed appearance choices already available in the current subset
4. Re-preview the component after edits.
5. Hand off generated React/CSS output or a reviewed FrameLab source file for engineering use.

The designer should not have to see:

- `.fl` source unless they choose an advanced view
- diagnostics JSON
- parser or validator terminology
- AST or IR concepts
- compiler stage boundaries

## What Stays Internal

The system may internally do all of the following:

- turn the prompt or brief into a generation prompt
- produce a first-pass `.fl` draft
- run `check` and `build`
- use diagnostics-guided repair when the first draft overreaches
- generate preview artifacts from compiled output
- preserve the reviewed `.fl` source as the real authored input

These internal steps should stay behind the workflow boundary unless an advanced user explicitly wants them.

## Roles In The Workflow

### Designer

The designer provides:

- a prompt or structured brief
- visual feedback on the preview
- structural refinement decisions in design-oriented language
- approval that the output is ready for handoff

### AI

AI is responsible for:

- converting the brief into a first-pass FrameLab draft
- applying repair when internal validation finds unsupported output
- translating design-oriented refinement requests back into valid FrameLab source

AI should stay constrained to the current supported subset.

### Design Tool

The design tool participates as the familiar refinement surface.

In the first workflow, this can be as simple as:

- a structured brief form
- a visual preview with editable labels for sections, variants, and slots
- optional Figma-adjacent or bridge-based structure hints where available

The design tool is not the compiler source of truth. It is the place where the designer sees and adjusts the component structure in familiar terms.

### Compiler

The compiler is responsible for:

- validating the FrameLab draft
- rejecting unsupported structure or syntax
- compiling the reviewed draft into production-shaped React/CSS artifacts

The compiler remains the system that determines whether the draft fits the supported subset.

## Minimal End-To-End Flow

### Step 1. Start From A Prompt Or Structured Brief

The designer starts with either:

- a prompt such as "Create a compact product card with thumbnail, category, title, price, and actions"
- or a small structured brief with fields like component name, primary text fields, variant axis, and slot regions

The brief should use design language, not FrameLab language.

### Step 2. AI Generates A First-Pass Draft

AI converts the brief into a `.fl` draft internally.

The designer sees:

- a generated component preview
- a short summary such as component name, variants, and editable content fields

The designer does not need to see the `.fl` source or validation details.

### Step 3. Preview The Result

The system builds a visual preview from compiler output.

The preview should reflect the current emitted subset only. In this milestone that means:

- static or token-backed visual styling
- stack and surface-based layout
- text content
- slot placeholders
- supported variant changes

The preview should not imply unsupported behavior such as native input semantics or runtime motion.

### Step 4. Refine Structure In A Familiar Design-Oriented Step

The designer refines the component using design-facing controls or language, for example:

- "move actions below the price"
- "add a small badge next to the title"
- "make the ghost button variant lighter"

Internally, AI translates those adjustments into valid FrameLab source and re-runs validation as needed.

If validation fails, the system should try repair internally and return an updated preview rather than exposing compiler errors directly.

### Step 5. Hand Off Production-Shaped UI

When the designer accepts the result, the workflow hands off:

- generated React component output
- generated CSS Module output
- shared token CSS output when applicable
- optionally the reviewed FrameLab source for engineering ownership

This handoff is production-shaped because it matches the compiler's current output model, not because it guarantees zero engineering review.

## Concrete Example Flow

### ProductCard Example

Designer brief:

- component: `ProductCard`
- goal: a navigable product card for a product grid
- content fields: category, title, price
- regions: thumbnail at the top, actions at the bottom
- tone: premium but compact

What happens next:

1. AI creates a first-pass `ProductCard` FrameLab draft using the current supported subset.
2. The system compiles it and shows a preview with:
   - a card surface
   - a thumbnail slot region
   - a vertical text stack for category, title, and price
   - an actions slot region
3. The designer refines the structure in design language:
   - "add a small badge next to the title"
   - "keep actions pinned below the text block"
4. Internally, AI updates the FrameLab draft, the compiler rechecks it, and the preview refreshes.
5. When accepted, the system hands off compiled `ProductCard.tsx`, `ProductCard.module.css`, and `tokens.css`.

What the designer sees:

- the brief
- the preview
- editable design-facing structure and content controls
- the final handoff package

What stays internal:

- the `.fl` source edits
- validation failures and repairs
- any diagnostics JSON used to keep the draft inside the supported subset

## Scope Limits For This Workflow

This first workflow only covers components that fit the current emitted subset well, especially:

- `surface`
- `stack`
- `text`
- `slot`
- `accessible`
- `intent`
- `variant`
- token-backed styling

It is a better fit for current examples like:

- `Button`
- `ProductCard`
- `NavItem`
- `Toast`

It is a weaker fit for anything that suggests unsupported behavior, especially:

- native form-control behavior
- animation or runtime motion systems
- rich application state beyond the current emitted subset
- broad page composition workflows

## Product Boundary

The first designer-facing workflow should present FrameLab as:

- a constrained component authoring system
- AI-assisted at draft and repair time
- preview-first from the designer perspective
- compiler-backed for production-shaped output

It should not present FrameLab as:

- a full design editor
- a full Figma replacement
- a no-review code generation promise
- support for the full frozen language surface
